# Demo Script 3min

Keep the `Mixed real/fallback` badge visible in the opening shot and mention it
before describing sponsor integrations.

0:00-0:25 - Problem. Software projects have useful work that is too small,
vague, or coordination-heavy to package. Agent work makes the trust problem
worse unless a system can prove what was funded, claimed, verified, accepted,
and settled.

0:25-0:55 - Project buys work. The project has a bounded budget policy. The
project-agent layer funds the Motoko TUI latency issue because it is measurable,
repo-bound, and verifier-backed. It declines vague, over-budget, or
under-specified work.

0:55-1:20 - Solver agents. Specialized profiles underwrite the bounty. Frontend
and CUDA profiles decline. The Python terminal/TUI solver claims because the
task matches its capability and expected margin.

1:20-1:55 - Verification. Show the contract digest, exact base and candidate
SHAs, changed files, protected verifier digest, receipt, and verdicts. Payment
is tied to the platform-owned verifier receipt, not a chat answer or a
candidate-controlled pipeline.

1:55-2:25 - Settlement. The accepted reward is split into an external transfer
portion and retained operating credit. Ledger entries are idempotent and
replay-safe. Retained credit can fund a second bounded bounty only through
trusted policy.

2:25-2:45 - Truth boundary. The bundle contains real Hermes executable evidence
and prior recorded-real Stripe transfer evidence. The fresh split Stripe
transfer, real GitHub lifecycle, Nemotron-backed Hermes decisions, and
OpenShell/NemoClaw execution are blocked or fallback until the exact external
setup is provided.

2:45-3:00 - Close. The contribution is the inspectable economic kernel: a path
from neglected task to verified work to exactly-once settlement, with honest
upgrade paths for every blocked sponsor component.
